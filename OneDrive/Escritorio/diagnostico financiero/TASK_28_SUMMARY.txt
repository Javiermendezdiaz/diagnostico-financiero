================================================================================
KEY FEATURES IMPLEMENTED
================================================================================

1. RETENTION POLICIES (Art. 5.1.e Storage Limitation)
   - Diagnoses: 12 months from creation
   - Open answers: 12 months from last update
   - Consent records: INDEFINITE (audit trail, never deleted)
   - Breach incidents: 3+ years (Art. 33-34 compliance)
   - Deletion requests: 30-day grace period (Art. 17 RTbF)

2. TWO-PHASE DELETION (Safety)
   - PHASE 1: Soft delete (marked deleted_at)
     * 7-day grace period for recovery
     * Data still in database but logically deleted
     * Audit trail preserved
   - PHASE 2: Hard delete (permanent removal)
     * After 7 days, record permanently removed
     * Execution timestamp recorded
     * No recovery possible

3. IMMUTABLE AUDIT TRAIL
   - JSON array in audit_log field
   - Entries: {timestamp, action, reason, actor}
   - Actions: scheduled, soft_deleted, hard_deleted, extended, etc.
   - No separate audit DB needed (embedded in entity)

4. BACKGROUND CLEANUP JOB
   - APScheduler-based cron (daily at 2 AM UTC)
   - Automatic missed-run catch-up
   - Configurable via CLEANUP_HOUR and CLEANUP_MINUTE env vars
   - Graceful startup/shutdown hooks

5. GDPR COMPLIANCE
   - Art. 5.1(e): Storage limitation enforced
   - Art. 17: Right to erasure via DELETION_REQUEST_30D
   - Art. 7(5): Burden of proof via audit_log
   - Art. 33-34: Breach retention (3 years)
   - Art. 32: Security via soft delete grace period
